CSS Proving Ground

CSS Proving ground is a special webpage where you can test your CSS. It intended to house all modern HTML5 elements you can spot in the wild in their natural habitat. So it could become a good help for any CSS check-list or even replace one completely.

To achieve such ambi­tious goal this page is filled with the real-life text (and not your usual “Lorem Ipsum”) littered with the HTML5 elements. Markup and page seman­tics both used as close to the current real-life cases as possible yet seman­tically correct.

Page Structure

In this section I will describe the structure of the most typical HTML5 web­page with <header> section for the website title and (optionally) description, ads and some other data; two navi­gation elements; <aside> element; one <article> section, where all the actual content of the page goes; and a footer with the copy­right note, con­tacts and such. Please note the first occurrence of the second-level heading just above this para­graph.

BODY Element

On this page <body> is the only wrapping element for the docu­ment sections like header, article and footer. For compara­tively simple pages this is the best choice. You can set page-level width and margins right to it. However, if you are going to use multiple side­bars, or other blocks, you would probably need some kind of <div id="wrapper">.

Also, take a note of the first occurrence of the third-level heading.

HEADER Element

<header> element of this page contains only title of the page in plain text. Style it as is or put your own graphics and/or custom HTML code. If you don't need it, just un­comment header { display: none; } CSS dirrec­tive to hide it.

Navigation

The role of navi­gation in this page is played by the two <nav> ele­ments. One of the class .top in the be­ginning of the page and one of the class .bottom in the bot­tom of the page, right be­fore the <footer>. Those ele­ments contain the table of con­tents struc­tu­red in ordered lists — nested one into the other.

Feel free to replace those ordered lists with your own navi­gation or turn the display of one or both of them on or off by commenting out or un­commenting respective { display: none; } dir­rec­tive.

ARTICLE Element

The main content of the page is encapsu­lated into the single <article> ele­ment. Keeping the content in a single ele­ment makes it seman­tically meaning­ful. This is much more use­ful than just ran­domly ID'd <div>:

  • Developers can easily style the con­tent of the <article> other way than the rest of the page
  • Search engines get the hint what text is the page's con­tent and what is navi­gational, aside, etc.
  • Users can reliably use Safari brow­ser Reader fea­ture and si­mi­lar soft­ware

FOOTER Section

This is simple. I put the copy­right notice and email add­ress here. If you want to style intri­cate, multi­column footer, put your divs into the <footer></footer>.

Text-level Semantics

In this part of the CSS Proving Ground the va­rious inline ele­ments will be tested. All those <i>, <strong> and some rarely used or new ones like <mark>. But as a star­ter point I want to focus your atten­tion on the essence of the World Wide Web — the links.

Links

Links is the glue that holds the Web toge­ther. With­out them it would be just a pile of HTML files. Even worse, search engines would hardly able to do their tre­men­dous jobs right with­out links.

Note that the first two links in the above para­graph are using HTTP pro­to­col and the third one is using its en­crypted ver­sion — HTTPS. With CSS you can style them diffe­rently if you want.

If your web­site has a lot of links to files, stored on some FTP ser­ver, you may want to add some vi­sual feature to such links to dis­tin­guish them from the common, HTTP ones.

Diffe­rent pro­to­cols are not the only thing that makes links diffe­rent. Their desti­nation could differ too. Some links can send you out­side, and some — through the diffe­rent parts of the same page, like back to the navi­gation or to another part of the article.

A little hint: clean your browser His­tory to test vi­sited and unvi­sited links pro­perly.

Emphasized and Strong Text

As the HTML5 specifi­cation suggests, <em> ele­ment should be used to gra­phi­cally show the text that is stressed. and the <strong> ele­ment should be used to mark the impor­tant parts of the text. Both these ele­ments are meant to push reader's atten­tion closer to the affected text. The diffe­rence bet­ween them, however, is that the former is tel­ling the user to read the marked text diffe­rently and the latter is tel­ling the user to put more atten­tion to the marked text.

Usually, the <em> ele­ment is styled with the font-style: italic;, and the <strong> element is styled with font-weight: bold; CSS directives. If you want to style some words with itali­cized or bold font, but do not intend to draw any seman­tically-driven atten­tion to them, use <i> and <b> res­pec­tively. Usage of <u> ele­ment as well as under­lining words that are not links should be avoided.

MARK Element

<mark> is the new ele­ment brought into the Hyper­Text with its fifth ver­sion. It could be used to high­light some text as you do it with the brightly colo­red mar­kers in the real life. Usually, it is styled with bright back­ground and maybe dar­ker text.

INS and DEL Elements

<ins> and <del> ele­ments are great if you want to show the pro­cess of work on your docu­ment. They are great for blog posts and published drafts, because you can visu­ally dis­tin­guish newly added parts from the ori­ginal text of the article and parts that ori­ginaly were the parts of the article but was consi­dered for removal due to in­accuracy or any other reason.

To make them even more useful, add a datetime attri­bute to them to help with ver­sion tracking and/or keeping a timeline of the document workflow.

Other Text-level Elements

There is a bunch of other ele­ments that could be used on a common web­page.

<abbr> is the HTML tag that comes in handy when you need to make abbre­vi­ations littered text more suit­able for the gene­ral audi­ence. It should go with the title="" attri­bute.

<kbd> and <samp> tags are often used to dis­play user input and com­puter out­put respec­tively. Like in the following example: If you will press Alt+F4, program will ask you Are you sure you want to close this window? Take a note that this example is en­closed in inline quota­tion tag — <q>.

Sometimes you will need to use shortened ordi­nal num­bers like 1st for “first”, 2nd for “second”, 3rd for the “third”, and so on. This is when you will need the <sup> ele­ment. Styling it with style, how­ever, could be a little bit tricky.

If you are one of those whose blood mostly con­sists of C8H10N4O2, and one of those who put C2H6O away until it is time to cele­brate the success, you will need the sub­script ele­ment — <sub>.

Code and Code Highlighting

The proper styling of code is very important for every­one who write about com­puter pro­gramming or sys­tem admi­nistra­tion. If you just paste chunks of code into your copy it would look horrible and will be barely read­able. Even the most beauti­ful code could be bro­ken by the use of variable width font and erratic inden­tation.

Here is example of Python script where the key­words, strings, vari­ables and comments are en­closed in the <span> ele­ments. Classes of those ele­ments are: .keyword, .string, .var, and .comment. The code is a simple Fibo­nacci gene­rator by joaquin.

# one space will separate better visually question and entry in console
numbers = raw_input("How many numbers would you like to display > ")    

# In a robust program you should use try/except to catch wrong entries
# Note the number you enter should be > 2: you print 0,1 by default
numbers = int(numbers)  

a, b = 0, 1        # tuple assignation
                   # note fibonaci is 0,1,1,2,3...

print a            # you can write this as print "%i\n%i" % (a, b)
print b            # but I think several prints look better in this particular case

for d in range(numbers - 2):  # you already printed 2 numbers, now print 2 less

    c = a + b 
    print c
    a, b = b, c    # value swapping
                   # A sorter alternative for this three lines would be:
                   # `a, b = b, a + b`
                   # `print b`

Lists

When writing for the Web, the thoughts you are going to bring to your audi­ence should be as much struc­tured as pos­sible. In elec­tronic world white­space is in abun­dance cause it doesn't take any space (nei­ther in sto­rage nor in net­work cables) at all. So if you have to say three dif­ferent things or more, say with the list.

Ordered Lists

  1. Ordered lists could be styled in many ways:
    1. As a decimal value
    2. As a latin alpha­bet symbol
      1. Uppercase
      2. Lowercase
    3. As a roman number
      1. Uppercase
      2. Lowercase
  2. You can set the start attri­bute to spe­cify from which num­ber (or letter) should item count go
  3. You can set re­versed attri­bute if you want your ordered list items num­bered from the greatest num­ber to the lowest one
  4. You can set specific number to a <li> ele­ment through the value attribute.

Unordered Lists

  • Unordered lists are much simpler than the ordered ones, but they a as well much more ver­satile:
    • You can use unordered list for building drop-down menus…
    • …picture galleries…
    • …search results lists…
    • …and many more
  • You can style the list items with diffe­rent list item markers or bullets:
    • circle
    • disc
    • square
    • you can even hide the marker away
  • Sometimes even pretty long para­graphs look good being the list items, but it is better to keep to the rule of thumb that the longest list item shouldn't be longer than a three or four lines of text
